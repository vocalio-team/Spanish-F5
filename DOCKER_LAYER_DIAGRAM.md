## How It Works

### The Golden Rule of Docker Caching

> **"Order matters: Put the most stable layers first, the most volatile layers last"**

Docker caches layers sequentially. When a layer changes:
1. ‚úÖ All layers **before** it remain cached
2. ‚ùå All layers **after** it must be rebuilt

### Example: Code Change Impact

**Before (bad order):**
```dockerfile
COPY . /app/              # Changes frequently ‚Üí invalidates everything below
RUN pip install deps      # ‚ùå Rebuilt (10 min)
COPY models/              # ‚ùå Rebuilt (2 min)
```

**After (good order):**
```dockerfile
RUN pip install deps      # ‚úÖ Cached (unchanged)
COPY models/              # ‚úÖ Cached (unchanged)
COPY src/                 # ‚ùå Rebuilt (only this - 50 sec)
```

## Best Practices Applied

### 1. Separate Dependencies from Code ‚úÖ
```dockerfile
# Copy requirements first (rarely changes)
COPY requirements-api.txt ./
RUN pip install -r requirements-api.txt

# Copy code last (changes frequently)
COPY src/ ./src/
```

### 2. Use Multi-Stage Builds ‚úÖ
```dockerfile
FROM base AS production
COPY --from=base /usr/local/lib/python3.10/dist-packages ...
COPY --from=base /app/src ./src/
```

### 3. Leverage .dockerignore ‚úÖ
```
tests/           # Don't need in production
docs/            # Skip documentation
*.pyc            # Skip compiled Python
__pycache__/     # Skip cache
```

### 4. Order Layers by Change Frequency ‚úÖ
```
1. System packages        (changes: never)
2. Python dependencies    (changes: rarely)
3. Pre-built models       (changes: occasionally)
4. Application code       (changes: constantly)
```
